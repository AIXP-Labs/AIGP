# Changelog — AINP

## v1.0.0 (2026-07-06) — initial release

- File family: `generation` / `generationreport` / `generationfeedback` / `generation_space` + `high_risk_types.v1.0.0.json` (versioned single source of truth, 3 artifact types + 17 risk tags with trigger bits and its own schema).
- Conformance rules: plan-side G1–G9, G11–G16 + FB1/SP1/SP2; report-side R1–R10; project package side P1–P11; three modes (default/strict/release); machine codes `AINP_[EWI]_<CODE>`.
- Reference tools: `tools/ainp_validate.py`, `tools/ainp_report_check.py`, `tools/ainp_release_check.py`, `tools/ainp_project_check.py` (stdlib-only).
- Examples: `file_family/` flat plans (incl. HSAW-gated likeness + mass-public landing page) + hash-locked report + feedback + space, plus `whitepaper_ainp/` complete document project package, `slugify_cli_ainp/` complete program project package and `aikp_navigator_aiap_ainp/` complete AIAP-package generation example with uppercase `AINP.md`, co-equal `ainp/` plan folder, optional `ainp/references/` protocol/template manifest and generated content folders; complete project packages declare `content_architecture`, and report/feedback/reference examples bind back to planned files/points or local integrity anchors; all pass default-mode validation and JSON Schema (`additionalProperties: false`); the flat whitepaper plan intentionally demos a G9 missing-binding WARN that escalates to strict/release FAIL (pinned in the example×mode matrix test).
- Added optional `ainp/references/` / `ainp.v1.0.0.reference_manifest` support for generation reference materials such as protocol snapshots, templates, briefs, style guides and research notes; P11 checks schema, local path containment and recorded local hashes without treating references as generated content or trust roots.
- Schema distribution: top-level `schemas/` is the single normative machine-readable source; `specification/schemas.md` and `specification/schemas_cn.md` index those files without copying them under `specification/`.
- Release engineering: added line-ending-normalizing `.gitattributes`, `pyproject.toml` console-script metadata, GitHub validation workflow, local `scripts/release_check.ps1`, Markdown link checker, documentation synchronization checker, and reference docs for release evidence, validator coverage, conformance rules and error codes.
- Tests: 79-test conformance suite incl. 36 fixture files, discovered `examples/*_ainp` package checks and example-project tests, project-package content-dir artifact placement, content-architecture path/file/point binding, standards-layer scope/alignment guard and feedback-loop alignment coverage, optional references/template manifest integrity, reference-manifest source-type schema conditionals, stdlib schema-keyword subset guard, program/AIAP example unittest coverage, unified CLI doctor coverage, doctor example-project and strict file-family discovery, delegated child-tool exit fail-closed checks, JSON-output path redaction on success and failure paths including encoded/fully-encoded unsafe-URI redaction and Windows/UNC/POSIX local paths with spaces/no extension, release-script POSIX private-path guard, release-script failure-output redaction and bad-root guard, doc-sync/Markdown-checker failure-output structure/redaction plus Markdown missing-target/local-machine-path/encoded-unsafe-scheme/reference-use/inline-code coverage, Markdown image-link, reference-link definition, matched-fence and HTML href/src/srcset/poster unsafe-target rejection, project hash-refresh/hash-sandbox/content-dir-required/content-dir-bounds/missing-artifact/duplicate-frontmatter/atomic-write/file-mode coverage, release-gate coverage including a high-risk likeness approval report, R3/SP2 sandbox escape coverage, parser-recursion guard, schema/tool closed-enum alignment, schema-location guard, example schema-profile guard, AINP.md frontmatter newline/delimiter guard and a pinned example×mode outcome matrix (79/79 green at release).

### Build-time completions relative to plan V5 (docs chain 12AINP_Protocol_Plan_V5_CN.md)

Recorded per the family discipline "deviations from the plan must be written down":

1. `PROVENANCE_STATUS = {unverified, retrieved, external_verify_required, externally_verified}` — enum was missing from V5 §0.
2. G2 sharpened: `acceptance_criteria` must be a **non-empty** array (a plan without criteria has no machine-checkable notion of "done").
3. Assurance matrix gained an explicit `static` row (satisfies all modes; release still requires G15's operational trio).
4. G9: URI-form binding targets (`://`) are classified as external verification and not stat-checked locally; `protocol` values `external`/`none` are skipped.
5. G7 realized as an exact-match banned-key set `{safe, verified, original, trusted, authentic}` scanned recursively.
6. G4 tag-consent mapping operationalized as "at least one `inputs[]` with `consent.required == true`".
7. Example layout: bindings demo consolidated in the whitepaper example (landing_page carries none) so high-risk examples stay release-clean.
8. **G6 high-risk definition extended** (found in build-time adversarial review): `risk_profile.risk_level ∈ {high, critical}` now triggers the approval gate on its own, matching the AIIP family convention (rule 11 gates on `governance.risk_level ∈ {high, critical}`). Previously a `critical` plan with no risk tag / non-`mass_public` scope / non-listed artifact_type could pass with `approval_required: false`. Fix landed in tool + spec §7/§10 + teeth fixture `critical_no_approval`.
9. G7 documented as a static exact-lowercase-key approximation (evadable by changing key names or case; genuine trust is a consumer/registry judgment) — no matcher widening, to avoid false positives on values like a tone of "safe".
10. **G4 rights-status path closed** (build-time adversarial review): `rights.status = consent_required` on an input now requires that same input to carry `consent.required: true` (code `AINP_E_G4_RIGHTS_CONSENT_NO_RECORD`, fixture `rights_consent_required_no_record`). Previously this path only produced a boundary WARN — declaring "consent needed" without a record is never permission to proceed.
11. Report-example verifier attribution corrected: the a1 structure check is attributed to the pipeline's own `structure-checker`, not to `ainp_report_check` — the reference checker validates report structure/gates (currently R1–R10) and never executes acceptance checks (honesty principle 2: the verifier field records who *actually* checked).

### Second adversarial audit (2026-07-06) — untrusted-input hardening

All findings below were confirmed by live probes before fixing, and each fix carries a teeth fixture:

12. **Crash class eliminated** (§8: hostile input must never crash a conforming tool). Non-object payloads (`generationfeedback`/`generation_space`/`generationreport` as string/list) raised `AttributeError`; deep nesting raised `RecursionError` both inside `json.load` (~100k levels) and inside the G7 trust scan (~2.4k levels). Fixes: payload-object gates (`AINP_E_FB1/SP1/R1_PAYLOAD_NOT_OBJECT`), parse-level `RecursionError` capture, and a post-parse iterative nesting guard `MAX_NESTING_DEPTH=150` (`AINP_E_NESTING_TOO_DEEP`). Top-level non-object inputs in version-diff mode gated too.
13. **Findings-flood DoS closed**: a plan with `inputs` set to a 200k-char string produced 200,000 findings (a 10 MB file could exhaust memory *within* the size cap). Container gates now emit one `AINP_E_CONTAINER_NOT_ARRAY` and refuse to iterate non-arrays (structure/inputs/acceptance_criteria/constraints/risk_tags/issues/generations); both tools additionally truncate at `MAX_FINDINGS=1000` with an explicit `AINP_E_FINDINGS_TRUNCATED` FAIL.
14. **G6 case/separator evasion closed**: `artifact_type: "Deepfake"` / `"deepfake "` / `"DEEPFAKE"` / `"deep fake"` / `"deep-fake"` / `"deep_fake"` variants must not slip past the high-risk type trigger with only a G3 WARN in default mode. High-risk type membership is now matched after case/separator normalization: whitespace, `_` and `-` separators are ignored (spec §7 normative; risk tags stay exact-match because unknown tags already hard-FAIL).
15. **Strict parsing**: duplicate object keys (parser-differential smuggling) and non-standard JSON literals (`NaN`/`Infinity`, accepted silently before) are now rejected at parse time (spec §8 items 6–7).
16. **R3 path-escape oracle closed**: in release mode the checker read and hashed *any* path the report named — including absolute paths outside the repository (confirmed against a system file). Artifact recompute now refuses absolute/escaping paths (`AINP_E_R3_PATH_ESCAPES`, spec §8 item 8) without reading them.
17. **R4 duplicates rejected**: the spec claimed "1:1 both directions" but duplicate `acceptance_results` for one criterion passed silently (`AINP_E_R4_DUPLICATE_RESULT`).
18. **R6 omission evasion closed**: the schema requires `overall`, but the checker silently skipped the R6 derivation when it was omitted (`AINP_E_R6_MISSING_OVERALL`).
19. **SemVer tightened to 2.0.0**: leading zeros (`01.2.3`) were accepted and build metadata (`+build`) was wrongly rejected; both fixed. Later audit aligned the JSON Schema with the validator and rejected prerelease numeric leading-zero identifiers such as `1.0.0-01`.
20. Honesty fixes: `high_risk_types` data-file failures are now explicitly fail-closed (missing file was already a FAIL, but the WARN text promised a graceful degradation that could never happen; a wrong-profile file now FAILs as `AINP_E_HIGH_RISK_TYPES_INVALID` instead of degrading on a WARN). `overall.warnings` as a string no longer substring-swallows warn-level failures. The G6 failure message now names all four high-risk triggers (it had lagged behind fix #8).

### Third adversarial audit (2026-07-06) — release-gate closure

21. **High-risk data shape fail-closed**: `high_risk_types` with the right profile but malformed internals (for example `artifact_types` as a string) no longer degrades into a G3 WARN and a missed G6 gate. The validator now checks non-empty `artifact_types[]`, non-empty `risk_tags[]`, unique normalized artifact types, unique tag ids and the three boolean trigger bits; malformed data raises `AINP_E_HIGH_RISK_TYPES_INVALID`. Added `schemas/high_risk_types-v1.0.0.schema.json`.
22. **R5 value-domain teeth**: `ainp_report_check.py` now rejects malformed `acceptance_results[].method`, non-object/bad `verifier`, and malformed evidence entries directly, not only through JSON Schema.
23. **R7 disclosure gates complete**: every true `disclosure_policy.*_required` switch now maps to a report-side recorded result. `machine_readable_metadata_required=true` requires `disclosure.machine_readable_metadata_present=true`; `content_credential_required=true` requires a present credential with `verification_status=externally_verified` (recorded evidence only, not independent credential validation).
24. **R9 approval evidence added**: if `plan.governance.approval_required=true`, the report must carry `governance_results.approval_gate` with method/evidence/verifier/limitations and `passed=true`. Approval-gate evidence/verifier shape failures are reported under R9-specific codes, not misfiled as R5 result-evidence failures. This closes the previous gap where high-risk reports could be `overall.conformant=true` without a separate approval evidence chain.
25. **Full release wrapper added**: `tools/ainp_release_check.py` runs bundled JSON Schema validation for plan/report/high-risk data, plan release validation, report release validation, artifact-hash recomputation through R3, disclosure/approval gates and `overall.conformant=true`. It outputs only `PASS release-structure-valid`, never legal/safety/trust claims.
26. **Project package layer added**: complete distributable AINP packages use `<name>_ainp/` with uppercase `AINP.md`, an `ainp/` AINP plan folder, and a `<name>/` project content folder. `AINP.md` is uppercase to match AIAP's `AIAP.md` convention and is checked by `tools/ainp_project_check.py` (now P1-P11). Project packages recompute artifact hashes against the package root sandbox, while standalone report checks still default to the report directory sandbox.
27. **Hash recompute sandbox hardened**: SP2 generation-space refs now refuse absolute or sandbox-escaping local paths before hashing, and R3/project package path checks use realpath/common-path logic to block symlink/junction escapes. Project package P6 also runs bundled JSON Schema checks for declared `feedback` and `space` side files before strict rule validation.
28. **Release JSON syntax gate tightened**: `ainp_release_check.py`, `check_doc_sync.py` and `scripts/release_check.ps1` now reject duplicate JSON object keys and non-standard literals at their own parse boundaries. Repository-wide syntax scans intentionally skip only `tests/fixtures/invalid/`, because those files are adversarial fixtures pinned by the test suite.
29. **Example profile drift guard added**: non-normative binding descriptors no longer use the top-level `schema` field, and the test suite now rejects unknown example `schema` tokens so examples cannot look formally schema-backed without a real AINP schema.
30. **Release wrapper recursion guard added**: `ainp_release_check.py` and project-wrapper child-output parsing now convert parser recursion into structured FAIL findings instead of allowing hostile deep JSON to surface as an unstructured crash.
31. **AINP.md frontmatter parser hardened**: project-package validation now accepts LF or CRLF line endings while requiring the opening/closing delimiter lines to be exact `---`, so Windows checkouts do not false-fail and malformed delimiters such as `---bad` do not pass.
32. **Release syntax gate widened**: `scripts/release_check.ps1` now compiles every repository Python source file outside generated/cache directories instead of relying on a fixed handwritten file list.
33. **Closed-enum drift guard added**: the test suite now compares JSON Schema enum values against the reference tool constants for plan, feedback, generation-space, report and project-package checks.
34. **Error-code reference drift guard added**: the documentation synchronization checker now verifies that machine-readable codes documented in `docs/reference/error-codes.md` are actually emitted by the reference tools; the stale `AINP_E_G6_APPROVAL_REQUIRED` example was corrected to `AINP_E_G6_MISSING_APPROVAL`.
35. **Mandatory boundary clarifications added**: the spec, schema descriptions and release evidence docs now state that the stdlib release-wrapper schema gate is a bundled-schema Draft 2020-12 keyword subset (full Draft behavior is tested with `jsonschema`), that content credential fields record external evidence rather than local C2PA/signature/trust-list verification, and that provenance/hash fields are integrity or evidence references rather than trust proofs.
36. **External transparency alignment fields added without a new trust layer**: plan `disclosure_policy` now supports `generator_metadata_required` and `watermark_required`; reports can record `generator` metadata and `disclosure.watermarks[]`; R7 fails only when the plan explicitly requires those records and the report omits them. High-risk tags gained `cbrn`, `self_harm` and `sexual_abuse_exploitation`. These are evidence records and risk triggers, not legal compliance or provider/watermark trust proof.
37. **Plan and content made co-equal in project packages**: AINP is now framed as a complete generation project protocol, not plan-only packaging. The honesty boundary still says `plan != artifact`, but `<name>_ainp/` now explicitly treats `ainp/` and `<name>/` as co-equal first-class folders. `tools/ainp_project_check.py` adds `AINP_E_P7_ARTIFACT_OUTSIDE_CONTENT_DIR`, failing project packages whose report artifacts resolve outside `AINP.md.content_dir` while still leaving malformed/escaping paths to R3/release sandbox checks.
38. **Bidirectional plan/content loop made mandatory for approved/active project packages**: `AINP.md.feedback` is required once a package is `approved` or `active`, `AINP.md` must include a `Feedback Loop` section, and `tools/ainp_project_check.py` checks that feedback `generation_id` matches the plan/report generation id (`AINP_E_P6_FEEDBACK_REQUIRED`, `AINP_E_P6_FEEDBACK_ID_MISMATCH`). This makes release-grade project packages explicitly bidirectional: plan guides content, content feedback flows back to the plan. The checker verifies loop wiring only; it does not claim the feedback is true, wise or complete.
39. **Content architecture made project-package mandatory and machine-bound**: standalone plans may omit `generation.content_architecture`, but when it is declared G16 validates the generated-content root, directories, concrete files and per-file points/requirements. Complete project packages require it through P10, reports can bind artifacts back to those files with `file_id`, and R10 requires every required content file to be covered by a matching artifact path. Project package P10 also requires planned content paths to resolve under `AINP.md.content_dir`, while P6 checks feedback issue `file_id`/`point_id` targets against the plan when feedback is declared. This keeps AINP as a generation-project protocol, not an AISOP execution graph, while making plan and content structurally co-equal.
40. **Lightweight standards layer added**: `specification/standards/` now contains `AINP_Standard.core.json`, `AINP_Standard.security.json` and `AINP_Standard.ecosystem.json` alongside the existing rules index. These files are machine-readable standard-library reference data for tooling and dashboards. They explicitly declare themselves non-executable, not AISOP flows, not AIAP programs and not runtime trust proofs.
41. **Example project package discovery guard added**: tests and `scripts/release_check.ps1` now discover `examples/*_ainp` packages and run `tools/ainp_project_check.py` for each package, so a newly added example package cannot remain outside the publication gate by omission.
42. **Project-local hash refresh helper added**: `tools/ainp_rehash.py` can check or refresh `generationreport.plan_ref.sha256`, report artifact hashes and local generation-space hashes. It is explicitly an authoring helper, not validation or trust proof; release still runs the normal validation gates afterward. The test suite also proves it refuses artifact paths that escape the project package sandbox before hashing.
43. **Hash-refresh missing-artifact guard pinned**: `tools/ainp_rehash.py --check --json` now has a dedicated regression test proving missing project artifacts return a controlled tool error instead of a vague hash/read failure.
44. Housekeeping: `ainp_rehash.py` unified its write/check/dry-run paths into a single scan (behavior-preserving; files are rewritten only under `--write`); the rehash README notes the project-package scope boundary (standalone file-family documents have no `AINP.md` entry point); the `nesting_too_deep` fixture dropped an incidental `content_architecture` block left over from when the field was mandatory (its 200-level chain lives in an unrelated field; teeth code unchanged).

### Fourth hardening pass (2026-07-08) — auxiliary-tool discipline

45. **Hash-refresh writes made atomic**: `tools/ainp_rehash.py --write` now writes JSON through a same-directory temporary file and commits with `os.replace()`, removing the only non-atomic in-place writer from the official helper set. A regression test forces replace failure and proves the original JSON remains intact with no `.tmp` residue.
46. **P8/error-code documentation made explicit**: P8 remains enforced by the delegated release gate, but the standards index and protocol table now state that it surfaces through release-family findings such as `AINP_E_RELEASE_NOT_CONFORMANT` rather than a dedicated `AINP_E_P8_*` code. `docs/reference/error-codes.md` now lists hostile-input/IO, release and strict/release escalation families.
47. **Auxiliary checkers tightened**: `check_markdown_links.py` now rejects undefined reference-style link labels, `check_doc_sync.py` narrows test-count detection so unrelated ratios do not false-fail while still catching stale suite counts, and `ainp_release_check.py` covers the remaining bundled-schema keyword subset used for release-side structural checks (`maxLength`, `maxItems`, numeric bounds, `multipleOf`, `patternProperties`, `propertyNames`, `prefixItems`).
48. **Assurance-matrix wording clarified**: the protocol now distinguishes control-point effects from final constraint verdicts, so `external_required` and `attested` evidence cannot be read as release-mode proof without an operational (`runtime | evidence_recorded | externally_verified`) control point. The same pass makes G4 consent evidence, G7/P9 self-trust boundaries, G5 strict escalation and AIIP rule 11 alignment explicit.
49. **High-risk report example completed**: `file_family/high_risk_likeness` now includes a recomputable PNG artifact, release-grade report and feedback record. The report demonstrates R7 content-credential recording and R9 approval-gate evidence while preserving the boundary that local tools do not prove consent validity, approver authority or credential trust.
50. **Feedback-loop example made non-happy-path**: the complete `whitepaper_ainp` package now uses a package-specific generation id and a structured `revise` feedback verdict, demonstrating content-to-plan review without implying automatic plan revision or external truth verification.
51. **Repository workflow hygiene tightened**: the GitHub validation workflow now runs the release gate on Python 3.10, 3.11 and 3.12, and the `spec-change` issue template now exists to match the contribution guide's governance process.
